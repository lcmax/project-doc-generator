# 项目文档生成器

根据项目代码自动生成完整的项目文档套件。

## 工作流程

### 1. 理解项目代码
- 分析项目目录结构
- 读取核心代码文件
- 理解领域模型、数据仓储、业务逻辑
- 识别设计模式和架构

### 2. 输出格式判断
根据用户提示词判断文档输出格式：
- **默认**：生成 Markdown（`.md`）文件
- **触发 DOCX**：当提示词中包含以下关键词（不区分大小写）时，生成 MS Word（`.docx`）文件：
  - `docx`
  - `word`
  - `.docx`

DOCX 生成策略：
1. 先按 Markdown 模板生成完整的 Markdown 内容
2. 使用 `pandoc` 将 Markdown 转换为 DOCX：
   ```bash
   pandoc <input>.md -o <output>.docx --toc --toc-depth=3
   ```
3. 转换完成后，仅保留 `.docx` 文件（删除中间 `.md` 文件）

pandoc 不可用时的回退策略：
- 输出 Markdown 文件
- 在文档头部添加提示：`⚠️ 未检测到 pandoc，已回退为 Markdown 格式。安装 pandoc 后可使用 DOCX 输出：winget install JohnMacFarlane.Pandoc`

### 3. 版本信息获取
- 检查项目是否使用 Git 版本控制
- 提取 Git 提交记录和提交次数
- 根据当前日期生成版本号：
  - **有 Git**：版本号格式为 `1.{yy}.{Mdd}.{git提交次数}`
    - `{yy}`：年份后两位（如 2026 年为 26）
    - `{Mdd}`：月份和日期（如 5月9日 为 509）
    - `{git提交次数}`：通过 `git rev-list --count HEAD` 获取
    - 示例：`1.26.509.1234`（2026年5月9日，第1234次提交）
  - **无 Git**：版本号格式为 `1.{yy}.{Mdd}.{hmm}`
    - `{hmm}`：小时和分钟（如 14:30 为 1430）
    - 示例：`1.26.509.1430`（2026年5月9日 14:30）

### 4. 作者信息获取
生成文档时，作者字段按以下三级优先级获取：
- **第1优先级 — 当前用户**：获取调用 Skill 的用户标识（PowerShell: `$env:USERNAME`，Linux/macOS: `whoami`）
- **第2优先级 — Git 提交作者**：从目标项目的 Git 最近一次提交中提取作者名（`git log -1 --format=%an`）
- **第3优先级 — 置空**：若以上均无法获取，作者字段留空（`作者：` / `Author: `）

### 5. 历史文档归档
生成新文档前，检查目标项目 `Doc/<项目名称>/` 目录下是否已存在 `cn/` 或 `en/` 子目录及文档文件：
- **若存在旧文档**：
  1. 读取旧文档头部信息中的版本号（`**版本**：` 字段）
  2. 在 `Doc/<项目名称>/` 下创建 `history/<旧版本号>/` 目录
  3. 将整个 `cn/` 和 `en/` 目录移动到 `history/<旧版本号>/` 下，保持原有子目录结构
- **若不存在旧文档**：跳过归档，直接进入下一步

归档后目录结构示例：
```
Doc/MyProject/
├── history/
│   └── 1.26.509.1234/        ← 旧版本号
│       ├── cn/
│       │   ├── 01-需求规格.md
│       │   └── ...
│       └── en/
│           ├── 01-Requirement Specification.md
│           └── ...
├── cn/                         ← 新生成的文档
│   └── ...
└── en/
    └── ...
```

### 6. 创建文档目录
在项目根目录下创建 `Doc/<项目名称>/cn/`（中文）和 `Doc/<项目名称>/en/`（英文）子目录（如已存在则检阅现有文档作为参考）

### 7. 生成文档套件

按顺序生成以下文档：

| 序号 | 中文文档 (Doc/cn/) | 英文文档 (Doc/en/) | 内容要点 |
|------|---------------------|---------------------|----------|
| 01 | 需求规格.md | 01-Requirement Specification.md | 功能概述、功能需求清单、业务规则需求、非功能需求 |
| 02 | 概要设计.md | 02-Overview Design.md | 系统架构、模块划分、类图关系、核心流程概要、依赖关系 |
| 03 | 详细设计.md | 03-Detailed Design.md | 每个方法的算法流程、分支逻辑、关键实现细节 |
| 04 | 数据库设计.md | 04-Database Design.md | 数据表结构、字段定义、表间关系、索引建议 |
| 05 | API文档.md | 05-API Documentation.md | 接口清单、入参出参、调用示例、异常场景 |
| 06 | 测试计划.md | 06-Test Plan.md | 单元测试用例、边界条件、并发测试、测试数据准备 |
| 07 | 部署手册与用户手册.md | 07-Deployment & User Manual.md | 环境要求、配置项、部署步骤、操作指南、常见问题 |

> **说明**：当输出格式为 DOCX 时，上述文件扩展名由 `.md` 变为 `.docx`（如 `01-需求规格.docx`、`01-Requirement Specification.docx`）。

### 8. 版本差异对比
新文档生成后，若 `history/` 目录下存在旧版本文档，则逐文件对比新旧文档内容，并在每个新文档末尾追加差异日志章节：

- 读取 `Doc/<项目名称>/history/<旧版本号>/` 下的对应文档
- 逐文件对比新旧文档的内容差异
- 在新文档末尾追加差异日志，按新增/修改/删除分类记录

差异日志格式（中文文档）：
```markdown
---

## 版本差异（v1.26.509.1234 → v1.26.510.1250）

### 🟢 新增
- 新增"四、性能监控"章节

### 🟡 修改
- 修改"二、功能需求清单"中模块B的描述

### 🔴 删除
- 删除"三、业务规则需求"中规则C
```

差异日志格式（英文文档）：
```markdown
---

## Version Diff (v1.26.509.1234 → v1.26.510.1250)

### 🟢 Added
- Added "4. Performance Monitoring" section

### 🟡 Modified
- Modified Module B description in "2. Functional Requirement List"

### 🔴 Removed
- Removed Rule C from "3. Business Rule Requirements"
```

特殊情况处理：
- **无差异**：差异日志注明"与上一版本无差异"
- **历史目录缺少对应文档**：标记为"全新文档"，不进行对比
- **无历史版本**：跳过差异对比步骤

### 9. 版本管理

#### 版本号规则
- **有 Git 版本控制**：版本号格式为 `1.{yy}.{Mdd}.{git提交次数}`
  - `{yy}`：年份后两位（如 2026 年为 26）
  - `{Mdd}`：月份和日期（如 5月9日 为 509）
  - `{git提交次数}`：通过 `git rev-list --count HEAD` 获取
  - 示例：`1.26.509.1234`（2026年5月9日，第1234次提交）
- **无 Git 版本控制**：版本号格式为 `1.{yy}.{Mdd}.{hmm}`
  - `{hmm}`：小时和分钟（如 14:30 为 1430）
  - 示例：`1.26.509.1430`（2026年5月9日 14:30）

#### Git 提交记录提取
- 使用 `git rev-list --count HEAD` 获取总提交次数
- 使用 `git log --pretty=format:"%h - %s (%an, %ar)" --date=short` 获取最近提交记录
- 提交记录用于生成变更日志和版本历史

#### 变更日志格式
在每个文档头部包含变更日志：

```markdown
## 变更日志

| 版本 | 日期 | 作者 | 变更内容 |
|------|------|------|----------|
| 1.26.509.1234 | 2026-05-09 | <动态获取> | 初始版本 |
| 1.26.510.1250 | 2026-05-10 | <动态获取> | 新增XXX章节，修正XXX描述 |
```

### 10. 文档头部信息

每个文档必须包含：

```markdown
# 文档标题

**项目名称**：XXX项目
**作者**：<动态获取>
**日期**：2026-05-09
**版本**：1.26.509.1234

## 变更日志
...
```

> **说明**：作者字段通过"4. 作者信息获取"中定义的三级优先级动态获取，不再使用硬编码的固定作者名。
